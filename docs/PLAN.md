## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | Teamver |
| 분류 | AI 활용 차세대 교육 솔루션 |
| 타겟 사용자 | 국내 대학교 팀 프로젝트(팀플)를 수행하는 대학생 |
| 핵심 가치 | 팀 구성 → 역할 설계 → 진행 추적 → 기여도 시각화까지 하나의 플로우로 완결 |
| 사용 AI 모델 | Gemini 2.5 Flash (Google AI Studio 무료 티어) |
| 개발 환경 | Cursor (AI 페어 프로그래밍), Claude (기획 및 프롬프트 설계) |

## 2. 문제 정의

### 2-1. 대상 사용자

- 국내 대학교에서 학기 단위 팀 프로젝트를 반복적으로 수행하는 대학생 전반
- 학기마다 새로운 팀 구성이 반복되며 역할이 유동적인 환경의 학습자

### 2-2. 핵심 페인 포인트

| # | 페인 포인트 | 구체적 상황 |
|---|---|---|
| 1 | 역할 불명확 | 팀 구성 직후 누가 무엇을 맡을지 정하는 데 1~2시간 소요, 갈등 발생 |
| 2 | 무임승차 | 기여도 측정 수단이 없어 불공정한 결과가 학기마다 반복됨 |
| 3 | 툴 진입 장벽 | Notion, Jira 등 기존 협업 툴은 2~8주짜리 팀플 대비 세팅 비용이 지나치게 큼 |
| 4 | AI 활용 공백 | 팀플에 AI를 적극 활용하고 싶지만 어떻게 녹여야 할지 알 수 없음 |

### 2-3. 기존 솔루션의 한계

기존 AI 협업 툴(Asana AI Teammates, ClickUp AI, Monday.com AI 등)은 모두 기업·직장인 대상으로 설계되어 있으며, 학생 팀플의 맥락을 해결하지 못한다.

| 비교 항목 | 기업 프로젝트 툴 | 학생 팀플 현실 |
|---|---|---|
| 프로젝트 기간 | 수개월 ~ 수년 | 2 ~ 8주 |
| 팀 구성 | 고정, 역할 명확 | 매 학기 변경, 역할 유동적 |
| 최대 고통 | 리소스·일정 관리 | 무임승차 + 역할 갈등 |
| 툴 학습 비용 | 감당 가능 | 툴 세팅하다 과제 기간 종료 |

> 한국 대학가에서 팀플 비중과 횟수는 지속 증가 추세이며,
> 졸업 후 기업이 요구하는 협업 역량과 직결되는 영역임에도
> 이를 전용으로 지원하는 도구가 국내에 존재하지 않는다.

## 3. 솔루션 설계

### 3-1. 핵심 기능

#### [ 기능 1 ] AI 킥오프 설계자

- 팀원 이름, 스킬셋, 성향, 프로젝트 목표, 기간을 입력
- Gemini가 최적 역할 분배안 + 주차별 마일스톤을 자동 생성
- `AI 팀원` 추가 가능 — 리서처, 문서 작성자 등 역할과 포지션 직접 설정
- 호출 시점 : 프로젝트 최초 생성 시 1회

#### [ 기능 2 ] 기여도 추적 보드

- 각 팀원이 본인의 작업 내용, 메모, 완료 항목을 직접 입력
- 일일 할당치 달성 여부 자동 체크 및 타임라인 시각화
- 누가 무엇을 언제 했는지 팀 전체가 투명하게 공유
- 추가 API 호출 없이 DB 기반으로 동작

#### [ 기능 3 ] AI 주간 리뷰

- 팀원이 `주간 리뷰` 버튼 클릭 시 선택적으로 호출
- 이번 주 전체 팀 기여 내용 요약 → 진행 상황 진단 + 다음 주 우선순위 재조정
- 정체 구간, 역할 충돌, 일정 지연 위험을 AI가 선제적으로 감지

#### [ 공통 ] 공동 목표 대시보드

- 팀 전체가 실시간으로 보는 단일 화면
- 목표, 진행률, 마감일, 역할 현황 한눈에 확인
- 모바일·데스크톱 모두 접속 가능한 반응형 웹

### 3-2. 사용자 플로우

```
팀원 초대 및 정보 입력
        |
        v
AI 킥오프 설계자 (Gemini 호출 #1)
  - 역할 분배안 생성
  - 마일스톤 생성
  - AI 팀원 설정
        |
        v
공동 목표 대시보드 활성화
        |
        v
각 팀원 일일 기여 입력 (API 호출 없음)
        |
        v
[선택] AI 주간 리뷰 (Gemini 호출 #2)
  - 진행 진단
  - 우선순위 재조정
        |
        v
프로젝트 완료 및 기여도 리포트 출력
```

## 4. 기대 효과

| 개선 항목 | 도입 전 | 도입 후 |
|---|---|---|
| 팀 세팅 시간 | 역할 논의에 1~2시간 | AI 킥오프로 10분 이내 완결 |
| 무임승차 억제 | 측정 수단 없음 | 기여 내용 기록·공유로 책임감 자연 증가 |
| AI 협업 역량 | 팀플에 AI 활용 방법 미지 | AI 팀원과 함께 운영하며 실전 경험 축적 |
| 툴 진입 장벽 | 세팅에 30분 이상 | 정보 입력 후 3분 안에 프로젝트 시작 |
| 졸업 후 연계 | 팀플 경험이 단절됨 | AI 협업 구조가 실무 역량으로 자연스럽게 이어짐 |

## 5. AI 활용 전략

### 5-1. 개발 단계 AI 도구

| 도구 | 용도 | 선택 이유 |
|---|---|---|
| Cursor | 코드 생성·리팩토링·디버깅 | Claude 기반 AI 페어 프로그래밍으로 혼자 개발의 속도 한계 극복 |
| Claude | 기획 검증, 문서 초안, 프롬프트 설계 | 이 기획서를 포함한 전 과정의 AI 협업 파트너 |

### 5-2. 서비스 내 AI 모델

**Gemini 2.5 Flash (Google AI Studio 무료 티어)**

선택 근거 :

- 무료 API(250 RPD)로 공모전 데모 수준의 완성도 달성 가능
- 1M 토큰 컨텍스트 윈도우로 팀 전체 프로젝트 맥락을 단일 호출에 포함 가능
- 비싼 모델 없이 정교한 프롬프트 설계만으로 동등한 결과를 내는 것 자체가 `AI 효율적 활용`의 증거

## 7. 시스템 아키텍처

```
[클라이언트 - React]
        |
        | HTTPS
        v
[백엔드 서버 - Node.js]
  - API 키 환경변수로 보호
  - 프롬프트 조립 및 요청 처리
  - 응답 JSON 파싱 및 검증
        |
        |---> [Supabase DB]
        |       - 팀 정보
        |       - 역할·마일스톤
        |       - 기여 로그
        |
        |---> [Gemini 2.5 Flash API]
                - 킥오프 에이전트
                - 주간 리뷰 에이전트
```

> API 키는 서버 환경변수에만 존재하며 클라이언트에 절대 노출되지 않는다.
> GitHub 저장소에는 `.env.example`만 포함하고 실제 키는 `.gitignore` 처리한다.

## 9. 토큰 최적화 및 유지보수 전략

### 9-1. 토큰 최소화

- **JSON 출력 강제** — system prompt에 JSON 스키마를 명시하여 불필요한 자연어 설명 텍스트 제거
- **선택적 호출 설계** — 킥오프(1회)와 주간 리뷰(선택)로만 API 호출을 제한. 실시간 채팅형 AI 미사용
- **컨텍스트 압축** — 기여 로그를 DB에서 집계된 요약 형태로만 전달, 원문 전체 전송 방지

### 9-2. 유지보수성

- **프롬프트 파일 분리** — 모든 system prompt를 `/prompts` 폴더에 `.txt` 파일로 버전 관리
- **AI 협업 문서 포함** — 본 기획서 및 에이전트 명세, 프롬프트 설계 과정을 `/docs`에 함께 커밋
- **모델 교체 용이성** — Gemini SDK 호출부를 별도 service 레이어로 추상화하여 모델 변경 시 단일 파일만 수정

### 9-3. 재현성

- **temperature 0.3 고정** — 역할 분배·마일스톤 생성은 창의성보다 일관성이 중요하므로 낮은 값 설정
- **few-shot 예시 내장** — 동일 입력에 대해 유사한 출력이 나오도록 system prompt에 예시 2~3개 포함
- **응답 검증 로직** — JSON 파싱 실패 시 fallback 처리 및 에러 로깅으로 재현 불가 상황 사전 차단
