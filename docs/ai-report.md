## 1. 기획

### ■ 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

**[ 대상 사용자 ]**

국내 대학교에서 학기 단위 팀 프로젝트(팀플)를 반복적으로 수행하는 대학생 전반.
특히 매 학기 새로운 팀이 구성되며 역할이 유동적인 환경에서 학습하는 학습자를 핵심 타겟으로 설정한다.

**[ 핵심 페인 포인트 ]**

| # | 페인 포인트 | 구체적 상황 |
|---|---|---|
| 1 | 역할 불명확 | 팀 구성 직후 누가 무엇을 맡을지 정하는 데 1~2시간 소요, 갈등 발생 |
| 2 | 무임승차 | 기여도 측정 수단이 없어 불공정한 결과가 학기마다 반복됨 |
| 3 | 툴 진입 장벽 | Notion, Jira 등 기존 협업 툴은 2~8주짜리 팀플 대비 세팅 비용이 지나치게 큼 |
| 4 | AI 활용 공백 | 팀플에 AI를 적극 활용하고 싶지만 어떻게 녹여야 할지 알 수 없음 |

**[ 기존 솔루션의 한계 ]**

Asana AI Teammates, ClickUp AI, Monday.com AI 등 기존 AI 협업 툴은 전부 기업·직장인 대상으로 설계되어 있다. 학생 팀플은 기간 2~8주, 역할 유동적, 무임승차가 최대 고통이라는 점에서 기업 툴이 이 맥락을 해결하지 못하며, 이를 전용으로 지원하는 도구가 국내에 존재하지 않는다.

---

### ■ 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

**[ 솔루션 개요 ]**

Teamver는 학생 팀플 전용 AI 협업 설계 도구다. 팀 구성 → 역할 설계 → 진행 추적 → 기여도 시각화까지 하나의 플로우로 완결하며, AI를 팀원으로 프로젝트에 포함시켜 실질적 협업 역할을 수행하게 한다.

**[ 기능 1 ] AI 킥오프 설계자**

- 팀원 이름, 스킬셋, 성향, 프로젝트 목표, 기간 입력
- Gemini가 최적 역할 분배안 + 주차별 마일스톤 자동 생성
- AI 팀원 추가 가능 — 리서처, 문서 작성자 등 역할과 포지션 설정
- 호출 시점 : 프로젝트 최초 생성 시 1회

**[ 기능 2 ] 기여도 추적 보드**

- 각 팀원이 본인의 작업 내용, 메모, 완료 항목을 직접 입력
- 일일 할당치 달성 여부 자동 체크 및 타임라인 시각화
- 누가 무엇을 언제 했는지 팀 전체가 투명하게 공유
- 추가 API 호출 없이 DB 기반으로 동작

**[ 기능 3 ] AI 주간 리뷰**

- 팀원이 주간 리뷰 버튼 클릭 시 선택적으로 호출
- 이번 주 전체 팀 기여 내용 요약 → 진행 상황 진단 + 다음 주 우선순위 재조정
- 정체 구간, 역할 충돌, 일정 지연 위험을 AI가 선제적으로 감지

**[ 공통 ] 공동 목표 대시보드**

- 팀 전체가 실시간으로 보는 단일 화면 — 목표, 진행률, 마감일, 역할 현황
- 모바일·데스크톱 모두 접속 가능한 반응형 웹

### ■ 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

| 개선 항목 | 도입 전 | 도입 후 |
|---|---|---|
| 팀 세팅 시간 | 역할 논의에 1~2시간 | AI 킥오프로 10분 이내 완결 |
| 무임승차 억제 | 측정 수단 없음 | 기여 내용 기록·공유로 책임감 자연 증가 |
| AI 협업 역량 | 팀플에 AI 활용 방법 미지 | AI 팀원과 함께 운영하며 실전 경험 축적 |
| 툴 진입 장벽 | 세팅에 30분 이상 | 정보 입력 후 3분 안에 프로젝트 시작 |
| 졸업 후 연계 | 팀플 경험이 단절됨 | AI 협업 구조가 실무 역량으로 자연스럽게 이어짐 |

## 2. AI 활용 전략

### ■ 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

**[ 개발 단계 도구 ]**

| 도구 | 용도 | 선택 이유 |
|---|---|---|
| Cursor | 코드 생성·리팩토링·디버깅 | Claude 기반 AI 페어 프로그래밍으로 1인 개발의 속도 한계 극복 |
| Claude | 기획 검증, 문서 초안, 프롬프트 설계 | 이 리포트를 포함한 전 기획 과정의 AI 협업 파트너 |

**[ 서비스 내 AI 모델 ]**

Gemini 2.5 Flash (Google AI Studio 무료 티어)

선택 근거 :

- 무료 API(250 RPD)로 공모전 데모 수준의 완성도 달성 가능
- 1M 토큰 컨텍스트 윈도우로 팀 전체 프로젝트 맥락을 단일 호출에 포함 가능
- 고비용 모델 없이 정교한 프롬프트 설계만으로 동등한 결과를 내는 것 자체가 AI 효율적 활용의 증거

### ■ 각 AI 도구별 활용 전략을 작성해주세요.

**[ Cursor — 개발 에이전트 전략 ]**

- 페르소나 설정 : 풀스택 주니어 개발자. 한국어 요구사항을 코드로 변환
- 활용 범위 : React 컴포넌트 생성, Supabase 연동 로직, API 라우팅, 반응형 UI
- 제약 사항 : 보안 관련 로직(API 키 관리, 인증)은 반드시 직접 검토 후 적용

**[ Gemini 2.5 Flash — 서비스 내 에이전트 구성 ]**

에이전트 1 : 킥오프 설계 에이전트

```
페르소나    : 10년 경력의 프로젝트 매니저
호출 시점   : 프로젝트 최초 생성 시 1회
temperature : 0.3

Input  : 팀원 정보 JSON + 프로젝트 목표 + 기간
Output : 역할 분배안(JSON) + 마일스톤 배열(JSON) + AI 팀원 설정값
```

에이전트 2 : 주간 리뷰 에이전트

```
페르소나    : 팀 퍼실리테이터 겸 리스크 감지자
호출 시점   : 팀원이 주간 리뷰 버튼 클릭 시 (선택적)
temperature : 0.3

Input  : 이번 주 전체 팀원 기여 로그 + 현재 마일스톤 달성률
Output : 진행 상황 진단 텍스트 + 다음 주 우선순위 재조정안(JSON)
```

데이터 흐름 :

```
사용자 입력
    |
    v
프론트엔드 (React)
    |
    v
백엔드 서버 (API 키 보호, 프롬프트 조립)
    |
    +---> Gemini API (에이전트 호출)
    |         |
    |         v
    |     JSON 응답 파싱 및 검증
    |
    +---> Supabase DB (결과 저장)
    |
    v
프론트엔드 렌더링
```

### ■ 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

**[ 토큰 최소화 ]**

- JSON 출력 강제 : system prompt에 JSON 스키마를 명시하여 불필요한 자연어 설명 텍스트 제거
- 선택적 호출 설계 : 킥오프(1회)와 주간 리뷰(선택)로만 API 호출을 제한. 실시간 채팅형 AI 미사용
- 컨텍스트 압축 : 기여 로그를 DB에서 집계된 요약 형태로만 전달, 원문 전체 전송 방지

**[ 유지보수성 ]**

- 프롬프트 파일 분리 : 모든 system prompt를 `/prompts` 폴더에 `.txt` 파일로 버전 관리
- AI 협업 문서 포함 : 기획서·에이전트 명세·프롬프트 설계 과정을 `/docs`에 함께 커밋
- 모델 교체 용이성 : Gemini SDK 호출부를 별도 service 레이어로 추상화하여 모델 변경 시 단일 파일만 수정

**[ 재현성 ]**

- temperature 0.3 고정 : 역할 분배·마일스톤 생성은 창의성보다 일관성이 중요하므로 낮은 값 설정
- seed 프롬프트 표준화 : 동일 입력에 대해 유사한 출력이 나오도록 system prompt에 few-shot 예시 2~3개 내장
- 응답 검증 로직 : JSON 파싱 실패 시 fallback 처리 및 에러 로깅으로 재현 불가 상황 사전 차단
